# Comprehensive Analysis of the Autoresearch Codebase

## 1. **What does this project do?**

**autoresearch** is an autonomous AI research framework that enables LLM agents to conduct hyperparameter tuning and architectural experimentation on a GPT language model **automatically and overnight**. Key characteristics:

- **Autonomous experimentation loop**: An AI agent modifies `train.py`, trains for exactly 5 minutes, evaluates performance, and decides whether to keep or discard the change
- **Fixed 5-minute time budget**: Training runs always complete in exactly 5 minutes (wall clock), making experiments directly comparable across different configurations
- **Single GPU, single file**: Designed for simplicity—only `train.py` is modified by the agent; runs on one NVIDIA GPU
- **Metric: Validation BPB** (bits per byte)—a vocab-size-independent evaluation metric where lower is better
- **Purpose**: Researchers provide instructions via `program.md` (markdown "skill"), then the agent runs ~100 experiments overnight while the human sleeps
- **Goal**: Find optimal model architecture, hyperparameters, and optimizer settings within the 5-minute time constraint

From README.md: *"The idea: give an AI agent a small but real LLM training setup and let it experiment autonomously overnight."*

#### **1. GPT Model (`GPTConfig` & `GPT` class)**

**Architecture features:**
- **Causal self-attention** with flash_attn_func (Flash Attention 3)
- **Multi-head key-value attention** (MQA variant): separate heads for Q vs K/V
- **Rotary embeddings** (RoPE) for positional encoding
- **Value embeddings** (ResFormer-style): learnable per-token value residuals with gated mixing
- **Alternating window attention pattern**: "SSSL" = Short, Short, Short, Long
  - Short = half context window (best-fit sliding window)
  - Long = full context window
  - Last layer always full attention
- **MLP with ReLU²** activation: `F.relu(x).square()` instead of standard GELU/ReLU
- **Per-layer residual scaling**: `resid_lambdas` (learnable multiplier on residual connections)
- **Per-layer x0 mixing**: `x0_lambdas` (learnable skip connection scaling)
- **Logit softcap**: `softcap * tanh(logits / softcap)` for numerical stability

**Configuration:**

**Weight initialization:**
- Token embeddings: Normal(mean=0, std=1.0)
- LM head: Normal(mean=0, std=0.001) — much smaller to stabilize training
- Attention & MLP weights: Uniform(±s) where `s = √3 * d_model^-0.5`
- Per-layer scalars initialized: `resid_lambdas=1.0`, `x0_lambdas=0.1`
- Value embedding gates: zeros (sigmoid(0)=0.5 → neutral scaling of 1.0)

#### **2. Optimizers (MuonAdamW)**

**Hybrid approach:** Different optimizers for different parameter types

**AdamW** (used for):
- Token embeddings (unembedding LR = 0.004, embedding LR = 0.6)
- Per-layer scalars: `resid_lambdas` (0.01× scalar_lr), `x0_lambdas` (0.5× scalar_lr)
- Value embeddings

**Muon** (used for):
- All 2D matrix parameters (attention & MLP projections)
- Feature: "Polar Express" orthogonalization via coefficients lookup table
- Feature: NorMuon variance reduction per-dimension
- LR scaling by aspect ratio: `lr * √(max(1, h/w))`

**Schedules:**
- **LR warmup**: Linear from 0 to full LR (configurable fraction of time budget)
- **LR cooldown**: Linear decay from full LR to final_lr_frac (default 0%)
- **Muon momentum**: Gradually increases from 0.85→0.95 over 300 steps
- **Weight decay decay**: Linearly decays to 0 as training progresses

**Compiled kernels:** Both `adamw_step_fused` and `muon_step_fused` use `@torch.compile` for GPU efficiency

```python

def evaluate_bpb(model, tokenizer, batch_size):
    """
    Bits per byte (BPB) metric:
    - Computes cross-entropy loss on validation set
    - Normalizes by target token byte lengths (UTF-8)
    - Excludes special tokens (byte_length=0)
    - Converts nats/byte to bits/byte
    
    BPB = sum(cross_entropy_loss) / (ln(2) * sum(target_bytes))
```

#### **1. Muon Optimizer with Polar Express Orthogonalization**

- Cautious weight decay (masked updates)

#### **2. Value Embeddings (ResFormer)**
- Learnable per-token value residuals
- Gated mixing: `v = v + gate * ve`
- Gate: `2 * sigmoid(linear(x))`
- Alternating per layer (not every layer)

#### **3. Per-Layer Scaling**
- `resid_lambdas`: scale on residual connections
- `x0_lambdas`: scale on skip connection from embedding

#### **4. Logit Softcap**
- Numerical stability: `softcap * tanh(logits / softcap)`
- Default softcap = 15

#### **5. Dynamic Learning Rate Schedules**
- Progress-based (not step-based)
- Warmup: linear from 0 to 1x
- Warmdown: linear decay over final fraction
- Muon momentum: gradually 0.85 → 0.95
- Weight decay: linearly decays to 0

#### **6. Attention Patterns**
- Sliding window attention (not full attention everywhere)
- Pattern: "SSSL" = alternating short/long windows
- Last layer always full attention

# **xla_rocm Project Analysis**

## **1. Project Overview**

**xla_rocm** is a **setup and patch distribution** project (not a fork) that enables running the Elixir machine learning stack (**Nx**, **EXLA**, **Bumblebee**) on AMD ROCm and NVIDIA CUDA GPUs with bleeding-edge hardware support.

- **Repository:** `https://github.com/chgeuer/xla_rocm`
- **Purpose:** Provide setup scripts that patch upstream XLA source, then build from scratch with GPU support
- **Key Problem Solved:** Pre-compiled EXLA binaries lack support for:
  - ROCm 7.2+ (ships `libamdhip64.so.7`, but XLA looks for `.so.6`)
  - Recent AMD architectures: `gfx1150` (Radeon 890M), `gfx1151` (Radeon 8060S)
  - NVIDIA Blackwell: `sm_120` (RTX 5070 / Blackwell)

---

## **2. Is This a Fork of EXLA?**

**No** — This is NOT a fork. It's a **downstream wrapper** around unmodified upstream EXLA/Nx.

- Uses the official hex packages: `{:nx, "~> 0.9"}`, `{:exla, "~> 0.9"}`, `{:xla, "~> 0.9.1"}`
- The **XLA source itself** (downloaded as part of the `:xla` hex dependency) is patched during build
- Provides **shell scripts** (`setup_rocm.sh`, `setup_cuda.sh`) that:
  1. Install system packages
  2. Patch the XLA C++ source code
  3. Build XLA from scratch with Bazel
  4. Compile EXLA NIF (native interface) with the patched XLA

### **1. HARNESS ADAPTER BEHAVIOUR (Required Callbacks)**

Five callbacks all adapters must implement:

```elixir
@callback id() :: atom()  # :claude, :gemini, :ghcopilot
@callback capabilities() :: Jido.Harness.Capabilities.t()
@callback run(RunRequest.t(), keyword()) :: {:ok, Enumerable.t(Event.t())} | {:error, term()}
@callback runtime_contract() :: RuntimeContract.t()
@callback cancel(String.t()) :: :ok | {:error, term()}  # Optional
```

**Capabilities flags (8):** `streaming?`, `tool_calls?`, `tool_results?`, `thinking?`, `resume?`, `usage?`, `file_changes?`, `cancellation?`

---

### **2. CLAUDE ADAPTER (jido_claude)**

**Binary wrapped:** `claude` (Anthropic)

**Pattern:** **TWO-AGENT** + Harness adapter
- Parent orchestrator agent spawns ClaudeSessionAgent children
- Each ClaudeSessionAgent manages one session + emits signals
- NOT stream-based; uses Jido.Agent pattern instead of pure Harness adapter pattern

**Connection mechanism:** **NONE** - Uses `ClaudeAgentSDK.query/2` directly
- Takes prompt → returns stream of SDK messages
- Mapper translates SDK messages → Harness events

**Capabilities:** streaming, tool_calls, tool_results, thinking (NO cancellation)

---

### **3. GEMINI ADAPTER (jido_gemini)**

**Binary wrapped:** `gemini` (Google)

**Pattern:** Pure Harness adapter (stateless, functional)
- Simple: RunRequest → `GeminiCliSdk.execute/2` → stream of events
- Mapper translates to Harness events

**Connection mechanism:** **NONE** - Stateless execution
- Similar flow to Claude but simpler (no two-agent pattern)

**Capabilities:** streaming, tool_calls, tool_results (NO thinking, NO cancellation)

---

### **4. GHCOPILOT (jido_ghcopilot) - HYBRID**

**Binary wrapped:** `copilot` (GitHub)

Has **THREE patterns**:

#### A. **Harness Adapter** (like Claude/Gemini)
- Wraps CLI in a Port
- Reads line-delimited JSON
- Maps via Mapper.map_line/2 → Harness events

#### B. **Server.Connection** (Stateful, different protocol)
```elixir
Connection.start_link(opts)
|> Connection.create_session(conn, opts)      # → {:ok, session_id}
|> Connection.send_prompt(conn, session_id, prompt)  # → {:ok, message_id}
|> Connection.subscribe(conn, session_id)     # → subscribe to events
# Receives: {:server_event, session_event}    # LSP-style JSON-RPC framing
```

#### C. **ACP.Connection** (Stateful, NDJSON protocol)
```elixir
ACP.Connection.start_link(opts)
|> ACP.Connection.new_session(conn, cwd, mcp_servers)
|> ACP.Connection.prompt(conn, session_id, prompt)
|> ACP.Connection.subscribe(conn, session_id)
# Receives: {:acp_update, session_update}     # NDJSON framing
```

**Capabilities:** ALL TRUE (most capable adapter) - including `cancellation?`

```elixir
             |> apply(sdk_function(), [request.prompt, options])
             |> Stream.flat_map(&mapper_module().map_event/1)

  end
end
```

- `id()` - atom identifier
- `capabilities()` - different capability sets
- `runtime_contract()` - env vars, CLI tools, install steps

- `sdk_function()` - query vs execute
- Mapper event types (SDK-specific message shapes)

```elixir
  @callback sdk_function() :: atom()  # :query or :execute

```
