## How It Works

**The core insight:** Instead of a human researcher manually tweaking neural network code, running experiments, and analyzing results — let an AI agent do all of that in an autonomous loop, 24/7.

### The Architecture (3 files)

| File | Role | Who edits it |
|------|------|-------------|
| `prepare.py` | **Infrastructure** — downloads data, trains tokenizer, provides dataloader + evaluation metric (`val_bpb`) | Nobody (fixed) |
| `train.py` | **The experiment** — GPT model, Muon+AdamW optimizer, training loop | The AI agent |
| `program.md` | **The "research org"** — instructions that tell the agent how to behave | The human |

### The Loop

```
forever:
  1. Edit train.py with an experimental idea
  2. git commit
  3. Run training (fixed 5-min budget)
  4. Read val_bpb metric
  5. If improved → keep commit, advance branch
     If worse → git reset, discard
  6. Log result to results.tsv
```

### What Makes It Elegant

1. **Fixed time budget, not fixed compute.** Every experiment is exactly 5 minutes of wall-clock training. This makes *any* change fairly comparable — whether you doubled model size, changed the optimizer, or rewrote the architecture. The agent optimizes *what's best for your specific GPU in 5 minutes*.

2. **Single metric, single file.** `val_bpb` (bits per byte) is vocab-size-independent, so even changing the tokenizer would be fair game. One file to edit means clean diffs and easy rollback.

3. **The "code" is the markdown.** The real programming happens in `program.md` — you're not writing Python, you're writing *instructions for a research organization*. The human programs the agent's strategy; the agent programs the neural network.

4. **Hill-climbing with git.** Git branches become the experiment tree. Keep improvements, revert failures. The commit history *is* the research log.

### Technical Highlights in `train.py`

- **Muon optimizer** for 2D matrix params (orthogonalized momentum via "Polar Express" approximation) + AdamW for everything else
- **Value Embeddings** (ResFormer-style) with input-dependent gating on alternating layers
- **Sliding window attention** pattern (`SSSL` = 3 short-window layers then 1 full-context layer)
- **ReZero-like** per-layer residual scaling (`resid_lambdas`, `x0_lambdas`)
- **Squared ReLU** activation (`relu(x)²`) instead of GELU/SwiGLU
- **Logit soft-capping** at 15 (Gemma-style)
- GC frozen after step 0 to avoid Python GC stalls

## How to Create Something as Inspiring

The recipe is a pattern you can apply to many domains:

1. **Pick a real but bounded task** — something measurable in minutes, not hours. LLM training in 5 min, image classification, code optimization, game playing, circuit design…

2. **Define one unambiguous metric** — `val_bpb` here. No multi-objective confusion. The agent needs a clear "better/worse" signal.

3. **Constrain the search space to one file** — the agent modifies a single, self-contained script. This keeps diffs reviewable and rollbacks trivial.

4. **Fix the evaluation, not the approach** — `prepare.py` is read-only. The goalposts don't move, but everything else is fair game.

5. **Write a `program.md`, not code** — your job shifts from "implementing ideas" to "designing the research process." You're programming the agent's methodology, curiosity, and judgment.

6. **Use git as your experiment tracker** — no MLflow, no W&B. Commits = experiments. Branches = research directions. The simplest possible infra.

7. **Make it autonomous** — the `NEVER STOP` instruction in `program.md` is key. The agent runs overnight, tries ~100 experiments while you sleep, and you wake up to a `results.tsv` of discoveries.

The genius is in what Karpathy *removed*: no configs, no distributed training, no experiment dashboards, no human-in-the-loop. Just an agent, a file, a metric, and time.

## Assistant — 2026-03-14T14:45:54.603000Z

Great question. They look similar on the surface — both are "agent loops that keep running" — but the core philosophy is quite different:

## Ralph Loop vs Autoresearch

| | **Ralph Loop** | **Autoresearch** |
|---|---|---|
| **Goal** | Complete a **checklist of tasks** (PRD) | Optimize a **single metric** (`val_bpb`) |
| **Termination** | Stops when all tasks are done | **Never stops** — runs until killed |
| **What the agent edits** | Any files needed to ship a feature | Exactly **one file** (`train.py`) |
| **Feedback signal** | "Are all checklist items done?" (binary) | Quantitative: did the number go down? |
| **On failure** | Re-try with fresh context | **Revert** (git reset) and try something else |
| **Loop driver** | External harness re-spawns the agent | The agent itself *is* the loop (via `program.md` instructions) |
| **Progress tracking** | `progress.txt` / `prd.json` managed by harness | `results.tsv` + git history managed by the agent |
| **Context strategy** | Fresh context each iteration to avoid drift | Fresh context is irrelevant — the metric is the memory |

### The Key Difference

**Ralph is a task-completion loop.** It has a finite goal (build feature X per the PRD), and the loop exists because agents run out of context or lose focus. The harness intercepts "I'm done" and says "no you're not, here's what's left." It's essentially a reliability wrapper.

**Autoresearch is a hill-climbing research loop.** There's no finish line. The agent proposes hypotheses, tests them empirically (5-min training run), and keeps or discards based on measured outcomes. It's closer to **evolutionary search** or **automated ML** than to task automation.

Put differently:
- Ralph asks: *"Did you finish the todo list?"*
- Autoresearch asks: *"Did you find something better?"*

Ralph solves a **known** problem. Autoresearch explores an **open** one.
